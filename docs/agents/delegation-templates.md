# 派工 Prompt 模板

複製對應模板，填空後派工。`〔〕`為必填空格；整段刪除標了（可選）的區塊沒關係，
但「目標／驗收條件／回報格式」三件套絕不可刪（理由見 `model-dispatch.md` 第 3 節）。

通則：

- subagent 是全新 context，完全不知道主對話發生過什麼。填空時把代名詞
  （「那個 bug」「剛才的方案」）全部展開成具體描述。
- **拿到 subagent 回報不等於完成**：交付給使用者前，仍要照
  `model-dispatch.md` 第 5 節做獨立驗證（程式碼跑測試＋實跑；檔案派 verifier read-back）。

---

## 1. 搜尋／定位（派 `Explore`，subagent_type: "Explore"）

```
在這個 repo 裡回答：〔要找什麼，一句話，例：文章刪除時音檔是在哪裡、由誰清除的〕

背景：〔為什麼要找，一句話——讓它判斷哪些結果相關〕
搜索廣度：〔medium ／ very thorough——跨多個目錄或命名不確定時用 very thorough〕
已知線索（可選）：〔已知相關的檔案、關鍵字、或已排除的位置〕

回報格式：直接回答問題＋每個結論附「檔案:行號」；如果找不到，列出你搜過哪些
位置與關鍵字，不要猜。不超過 15 行。
```

## 2. 實作（派 `implementer`）

```
目標：〔做什麼，例：POST /lookups 的每日全站上限改為可由環境變數 LOOKUP_DAILY_CAP 調整〕
動機：〔為什麼，例：目前寫死 500，正式環境需要不重建 image 就能調〕

相關位置：〔已知的檔案路徑，例：api/src/routes/lookups.ts、api/src/config.ts〕
約束：〔例：預設值維持 500；不得新增依賴；沿用既有 config 讀取模式〕
已排除的方案（可選）：〔例：不要做成 DB 設定表，討論過太重〕

驗收條件（全部達成才算完成）：
1. npm test 全綠（不設 DATABASE_URL）
2. npm run typecheck 全綠
3. 新行為有新測試：〔例：設 CAP=2 時第 3 次呼叫回 429〕
4. 〔其他可觀察行為〕

回報格式：結論／改動檔案清單（路徑:行號）／驗收指令與結果／未解事項，不超過 20 行。
不要 commit。
```

## 3. 重構（派 `implementer`）

```
目標：重構〔範圍，例：worker/src/processor.ts 的段落處理流程〕，
使〔改善目標，例：翻譯與 TTS 兩步驟可獨立重試〕。
動機：〔例：目前 TTS 失敗會連翻譯一起重跑，浪費 LLM 費用〕

不變式（重構後必須維持的外部行為）：
1. 〔例：jobs 表的狀態機不變：pending→processing→done/failed〕
2. 〔例：對外 API 回應格式不變〕

紀律：先跑一次 npm test 確認起點是綠的再動手；小步改、每步跑測試；
禁止為了讓測試通過而修改測試的斷言——測試紅了就是重構破壞了行為。

驗收條件：npm test＋npm run typecheck 全綠；〔如有：新增的不變式測試〕。
回報格式：結論／改了什麼與為什麼（3 行內）／改動檔案清單／驗收結果。不要 commit。
```

## 4. 研究（派 `general-purpose`）

```
研究問題：〔一句話，例：Fastify v5 的 onRequest hook 在 404 路由上是否觸發〕
動機與將做的決策：〔例：決定 rate-limit 要掛在哪一層〕

要求：
- 優先查官方文件與原始碼，其次才是部落格；每個結論附來源 URL。
- 區分「文件明說的」與「你推論的」，推論要標明。
- 查無就回報查無＋你查過哪裡，不要編造。（Claude Code／Anthropic API 的問題
  改派 claude-code-guide。）

回報格式：先給一句話結論（直接可用於決策），再列證據與來源。長篇整理落檔到
〔scratchpad 或 docs/ 路徑〕，回報只給路徑＋摘要。不超過 15 行。
```

## 5. 審查／驗收（派 `verifier`；高風險時呼叫加 model: "opus"）

```
待驗對象：〔檔案路徑清單，或「working tree 目前的未提交改動」〕
這次改動宣稱做到：〔只寫「應該是什麼」，不要寫實作過程，不要透露你希望的結論〕

驗收條件（逐條判定通過／不通過）：
1. 〔例：npm test 全綠（不設 DATABASE_URL），且新增測試覆蓋新行為〕
2. 〔例：docs/agents/*.md 中引用的路徑與指令全部真實存在〕
3. 〔例：沒有違反 CLAUDE.md 紅線的內容〕

另外請主動找：驗收條件沒覆蓋到、但會讓這個產出失敗的問題。

回報格式：第一行總判定；逐條判定＋證據（檔案:行號／指令輸出關鍵行）；
最後列驗收條件外的風險。不超過 25 行。
```

## 常見壞派工對照（自查用）

| 壞寫法 | 為什麼壞 | 修法 |
|--------|----------|------|
| 「修好那個測試」 | subagent 不知道「那個」是哪個 | 寫出測試檔路徑與失敗訊息 |
| 「確保程式品質良好」 | 無法判定，等於沒寫 | 換成可跑的指令與可觀察行為 |
| 「順便看看有沒有其他問題」 | 範圍無界，回報會爆炸 | 拆成獨立的審查派工 |
| 驗收條件＝「你覺得完成就好」 | 自由心證，弱模型會提早宣稱完成 | 每條驗收都要可機器判定 |
| 一個 prompt 派三件事 | 中途失敗全部重來，回報混雜 | 一個 agent 一件事，可並行派 |
