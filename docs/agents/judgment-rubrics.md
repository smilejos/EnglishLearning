# 判斷力 Rubric

把「需要判斷力」的時刻轉成可執行的判準。每條附正例（該這樣）與反例（不該這樣）。
拿不準時，照字面執行判準即可——這些規則寬鬆執行比自由發揮安全。

---

## 1. 何時升級模型

**判準**：出現以下任一訊號就升級（路徑見 `model-dispatch.md` 第 6 節），不要再調 prompt 重試同型號：

- 同一子任務驗收失敗達到門檻（haiku 一次、sonnet 兩次）。
- 修好 A 弄壞 B、再修 B 又弄壞 A——在兩個狀態間震盪。
- 問題涉及跨模組互動、併發、時序、資料一致性，而你只能描述症狀、說不出機制。

✅ 正例：sonnet 兩次修 worker 測試都失敗，第二次還讓 api 測試變紅 → 開新 opus agent，
附上兩次的改動 diff 摘要與失敗輸出，請它先解釋機制再動手。
❌ 反例：sonnet 第四次微調同一個 prompt 再派給 haiku，理由是「上次只差一點」——
差一點就是失敗，門檻已到。

## 2. 何時算真的完成

**判準**：以下全部成立才能對使用者說「完成」：

1. 驗收指令親自跑過且全綠（`npm test`＋`npm run typecheck`，加上任務指定的驗證）。
2. 新行為有對應的新測試，或已實跑觀察到新行為（打端點、開頁面）。
3. 使用者原始要求逐句對照過，沒有漏掉子項（「加篩選器**並且**後台同步」是兩件事）。
4. 沒有留下你知道但沒說的問題。

✅ 正例：「完成。lookups 上限改為環境變數，新增 2 個測試（`api/src/routes/lookups.test.ts:88`），
npm test 104→106 全綠，typecheck 綠，實測 `LOOKUP_LIMIT=2` 時第三次呼叫回 429。」
❌ 反例：「已修改完成，程式邏輯上應該可以正常運作」——「應該」就是沒驗，等於未完成；
或測試全綠但這次改的是前端行為、而測試根本沒碰前端。

## 3. 何時停下來問使用者

**判準**：只有這四類該問，其他自己決定並在回報中說明你選了什麼：

- 不可逆且影響真實資料／對外行為（動 5432 正式庫、刪東西、push、改對外 API 行為）。
- 使用者要求有兩種讀法，且做錯要重工超過半小時。
- 需求與紅線衝突（例如要求會呼叫真實 Gemini API）。
- 三次嘗試失敗、換路訊號也出現了（見第 4 條）。

✅ 正例：「這個 migration 會改正式庫的 paragraphs 表結構，我準備了 up/down 並在測試庫演練過，
要執行嗎？」
❌ 反例：「變數名用 `maxRetries` 還是 `retryLimit` 好？」——可逆、無歧義成本，自己挑一個；
也不要用「我先做 A，你若不同意再說」偷渡不可逆操作。

## 4. 方向錯了該換路（而非重試）的訊號

**判準**：出現任一訊號，停止重試，退回上一個決策點列出替代路線：

- 為了讓方案成立，需要的「例外處理」越改越多（第三個特例出現就是訊號）。
- 需要改動的檔案數量隨著理解加深不斷擴大（原估 2 檔，現在要動 8 檔）。
- 你開始想改測試來遷就實作，而不是改實作通過測試。
- 方案依賴一個你無法驗證的假設（「應該沒有其他地方用到這個欄位」——去查，查不動就是訊號）。

✅ 正例：改 audio 路徑第三次遇到「還有一處寫死」→ 停手，派 Explore 掃出所有寫死處，
發現該用反向代理統一處理，重新提案。
❌ 反例：測試一直紅，於是把斷言從「等於」改成「包含」讓它綠——這是把警報器拆掉，不是滅火。

## 5. 品質底線怎麼驗

**判準**：不同產出各有最低驗法，缺這一步不得交付：

- 程式碼 → 測試＋typecheck＋實跑受影響的那條路徑一次（`docs/agents/model-dispatch.md` 第 5 節）。
- 文件／設定檔 → 派 `verifier` read-back：內容完整、內部一致、路徑與指令實際存在。
- 資料操作 → 先在 5433 測試庫演練，數 row 數對帳，才碰正式庫（且要先問，見第 3 條）。
- UI 改動 → 至少 curl 或開頁面確認 render 成功，理想是截圖給使用者。

✅ 正例：改了 nginx 設定 → `docker compose up` 後 curl 8081/8082 確認 200，再看一眼
容器 log 沒有 warning。
❌ 反例：「設定檔語法我檢查過了沒問題」——語法對不等於行為對，要實跑。

## 6. 誠實條款：這套制度補不了的事

拆解、驗證、多樣本評審能把**執行品質**補到接近強模型；但以下兩類補不了，遇到就照寫的辦：

- **模糊題**（需求本身沒有客觀對錯，如「介面要更優雅」）：不要假裝有標準答案。
  做出 2–3 個具體差異明顯的方案（截圖或對照表）讓使用者選，選項本身就是你的產出。
- **品味與架構直覺**（兩個方案都能過驗收，長期優劣難量化）：明說「兩案皆可行，
  我的傾向與理由如下，但這屬於品味判斷，信心中等」。重大架構決策建議使用者
  自行審視或用一次高階模型（opus 甚至外部意見）覆核，不要讓 sonnet 假裝篤定。
- 查不到的事實（API 規格、版本行為）：派 `claude-code-guide` 或查官方文件；
  查不到就寫「查無，以下是假設」，**絕不編造**。
