# 制度檔案維護協議

管的是：`CLAUDE.md`、`docs/agents/*.md`、`.claude/agents/*.md`。
這些檔案是所有 session 的作業系統，改壞的影響是全域且持續的，所以更新有規矩。

## 1. 權限分級

**可以自行改（改完在當次回報中告知使用者即可）**：

- 事實修正：路徑、指令、行號、model 名稱與現實不符（先驗證新事實為真再改）。
- 「踩坑記錄」的追加（見第 3 節格式）。
- 錯字、排版、失效連結。

**先問使用者才能改**：

- 任何規則的新增、刪除、放寬或收緊（包括紅線、升降級門檻、派工判準）。
- 重寫或重組任何一個檔案的結構。
- 修改 `.claude/agents/*.md` 的 model／effort／tools 設定。
- 刪除或搬移任何制度檔案。

**絕不改**：`docs/agents/backup/` 下的備份、`diagnosis-2026-07-07.md` 與
`letter-to-future-sessions.md`（歷史文件，只讀不寫）。

## 2. 改動程序（自行改也要走）

規則等級的改動（新增／刪除／改寫規則）在提案前，先讀 `diagnosis-2026-07-07.md`
了解該規則的原始理由，提案時一併說明為何原理由不再成立。

1. 先備份：`cp 檔案 docs/agents/backup/檔名.YYYY-MM-DD.bak`（同日多次改動只需備份第一次）。
2. 改動。
3. 派 `verifier` read-back：確認改動後檔案內部一致、引用的路徑與指令仍真實存在、
   沒有和其他制度檔互相矛盾。
4. 在當次任務的收尾回報中，用一句話告知使用者改了什麼。commit 仍照 CLAUDE.md
   的 Git 流程走「滿意確認」，本協議不豁免。

## 3. 踩坑教訓寫回哪裡、用什麼格式

判斷歸屬：

- 這個專案特有的事實（指令、路徑、服務行為）→ `CLAUDE.md` 對應段落，一行以內。
- 派工／驗收／升級的流程教訓 → 對應制度檔（`model-dispatch.md` 等）末尾的
  「## 踩坑記錄」段落（沒有就新建）。
- 使用者偏好、跨 session 的工作脈絡 → 記憶區
  （`~/.claude/projects/-Users-jos-Documents-program-EnglishLearning/memory/`，
  依 harness 記憶格式寫，並更新 MEMORY.md 索引）。

踩坑記錄格式（一坑一條，三行以內）：

```
- YYYY-MM-DD｜情境：〔做什麼時〕｜坑：〔發生什麼〕｜以後：〔一句可執行的指示〕
```

寫入門檻：只記「下次還會遇到、且靠這一條就能避開」的坑。一次性的意外不記。

## 4. 何時精簡、怎麼精簡

- 觸發條件（任一）：`CLAUDE.md` 超過 120 行；任一制度檔的踩坑記錄超過 10 條；
  發現兩條規則說同一件事或互相矛盾。
- 做法：這是「規則的新增刪除」等級的改動 → **先問使用者**，提案內容為：
  哪些條目合併、哪些過時可刪、刪掉的內容摘要。獲准後才動手，動手前備份。
- 精簡的方向：把重複收斂成一條、把長解釋移到 `docs/agents/` 引用檔、
  把已內化為程式碼或測試的規則刪掉（例如某紅線已有自動防呆，就可降級為一句備註）。

## 5. 每季健檢（使用者提出「健檢制度檔」時執行）

1. 逐檔驗證所有路徑、指令、model 名稱仍然有效（派 `verifier`）。
2. 對照 `.claude/settings.local.json` 的新增 permission 歷史，看主對話是否仍在下場做
   該委派的事——是的話，回報哪條派工規則沒被遵守，討論是否要改寫得更明確。
3. 回顧踩坑記錄：已經不會再踩的刪掉（先問），反覆出現的升級成正式規則（先問）。
